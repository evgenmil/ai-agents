# Тестирование размера чанков (chunk_size)

Документ фиксирует результаты ручного A/B-теста параметров `RecursiveCharacterTextSplitter` и рекомендации для проекта.

**Дата:** 02.06.2026  
**Сырой лог диалогов:** [`test_chunk_size.md`](../test_chunk_size.md)

## Условия эксперимента

| Параметр | Значение |
|----------|----------|
| Сплиттер | `RecursiveCharacterTextSplitter` (`src/indexer.py`) |
| Единица измерения | символы (`chunk_size`, `chunk_overlap`) |
| Overlap | ~10% от `chunk_size` |
| Источник | PDF в `data/` (кредиты, вклады) |
| Retriever | `RETRIEVER_K=3` (без изменений) |
| Способ проверки | Telegram-бот, один и тот же набор из 3 вопросов на каждую конфигурацию |
| После смены параметров | перезапуск бота / переиндексация |

Протестированные конфигурации:

| Конфиг | `chunk_size` | `chunk_overlap` |
|--------|--------------|-----------------|
| A | 300 | 30 |
| B | 800 | 80 |
| C | 1500 | 150 |

## Тестовые вопросы

1. Как уменьшить размер аннуитетного платежа?
2. Как внести изменения в кредитный договор?
3. Какие есть вклады?

## Сводное сравнение ответов

| Вопрос | 300 | 800 | 1500 |
|--------|-----|-----|------|
| Аннуитетный платёж | Общий ответ: досрочное погашение без сокращения срока; упомянуты реструктуризация, срок, ставка | Конкретика из документа: **«Вариант 1»**, срок и число платежей не меняются; договор > 1 месяца | Максимум деталей: сумма **больше ближайшего платежа**, «Вариант 1», пересчёт ежемесячного платежа |
| Изменения в договоре | Реструктуризация, соглашение сторон, ИУ — без процедурных деталей | **Письменная форма**, подписи, исключения, уведомление за **5 календарных дней** | Снова акцент на реструктуризацию; слабее, чем при 800, по формальному оформлению изменений |
| Список вкладов | Другой набор названий («Детская копилка», «Доступная выгода» и т.д.) | Согласованный с PDF перечень («Лучший %», «Сбер Рядом», «СберВклад» и др.), развёрнутый | Близко к 800, но короче и обобщённее |

## Выводы по конфигурациям

### chunk_size=300

- Ответы по кредитам **общие**, без терминов из документов («Вариант 1»).
- По вкладам — **расхождение** с ответами при 800/1500 (другие названия продуктов).
- Вероятная причина: мелкие чанки + `k=3` → в контекст попадают менее релевантные фрагменты.

### chunk_size=800

- **Лучший баланс** на всех трёх вопросах: и досрочное погашение, и процедура изменения договора, и согласованный список вкладов.
- Достаточная детализация без избыточного «шума» в retrieval.

### chunk_size=1500

- Сильнее всего на вопросе про аннуитет (дополнительные условия по сумме погашения).
- На вопросе про изменение договора **уступает 800**.
- По вкладам сопоставим с 800, но ответ короче.

## Итоговая рекомендация

**Принять для проекта: `chunk_size=800`, `chunk_overlap=80`.**

Обоснование:

- Наиболее стабильное качество по смешанному корпусу (кредиты + вклады).
- 300 даёт худший retrieval по вкладам; 1500 не даёт стабильного выигрыша на всех типах вопросов.
- Overlap 10% — согласован с общей практикой и предыдущими рекомендациями.

Параметры зафиксированы в `src/indexer.py`.

## Ограничения теста

- Всего **3 вопроса**, без уточняющих («А по вкладу X?»).
- Не менялся `RETRIEVER_K` — связка «размер чанка × k» не исследовалась.
- Субъективная оценка по смыслу ответов, без автоматической метрики (hit rate, LLM-judge).
- Индексировались PDF; JSON `data/sberbank_help_documents.json` в этом прогоне не использовался.
