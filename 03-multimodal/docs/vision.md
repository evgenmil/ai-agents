Техническое видение проекта basic-ai-assistant
=============================================

Этот документ описывает минимальное техническое видение Telegram‑бота — персонального финансового советника на базе LLM. Цель — максимально простой MVP для учёта доходов и расходов через чат, без оверинжиниринга.

## 1. Технологии, зависимости и сборка

- **Язык**: Python 3.11 (минимальная поддерживаемая версия).
- **Управление зависимостями**: `uv` с использованием `pyproject.toml` и `uv.lock`.
- **Работа с LLM**: официальный клиент `openai`, настроенный через `base_url` и API‑ключ из переменных окружения. Один и тот же клиент работает с OpenRouter и с локальным Ollama (OpenAI‑совместимый endpoint `/v1`).
- **Structured output**: извлечение транзакций из текста — через `response_format` (JSON Schema) и Pydantic‑схемы.
- **VLM**: обработка фото чеков — vision‑запрос к модели из `MODEL_IMAGE` с тем же structured output.
- **Интеграция с Telegram**: `aiogram`, только режим `polling` для простоты (без вебхуков и дополнительного веб‑сервера).
- **Локальный запуск без Docker**: `make` вызывает `uv run` с основным entrypoint модуля приложения.
- **Локальный запуск в Docker**: `Dockerfile` на базе `python:3.11-slim`, внутри — установка `uv`, установка зависимостей и запуск того же entrypoint, что и локально.
- **Минимальность стека**: на этапе MVP не используются внешние БД, очереди и т.п.; транзакции и контекст диалогов хранятся в памяти процесса.

## 2. Принципы разработки

- **KISS и минимализм**
  - Каждая новая сущность/абстракция появляется только при реальной необходимости.
  - Предпочитаются простые, прямолинейные решения без лишних слоёв.
- **ООП с принципом 1 класс = 1 файл**
  - Каждый класс располагается в отдельном файле с говорящим именем.
  - Классы небольшие, с понятной и узкой зоной ответственности.
- **Разделение зон ответственности без оверинжиниринга**
  - Модуль/пакет для работы с Telegram (`aiogram` и обработка апдейтов).
  - Модуль/пакет для работы с LLM (structured output, vision).
  - Пакет `finance` — учёт транзакций и бизнес‑логика советника.
  - `FinanceManager` — ядро: контекст диалога, вызов LLM, сохранение транзакций, ответы пользователю.
- **Простота тестирования и отладки**
  - Логика по возможности отделена от конкретных клиентов (Telegram, HTTP), чтобы её можно было вызывать напрямую из кода или CLI.
  - Тесты не являются обязательными на первом шаге, но код пишется так, чтобы их можно было легко добавить.
- **Явная конфигурация**
  - Секреты и адреса внешних сервисов хранятся только в переменных окружения.
  - Переключение OpenRouter ↔ Ollama — сменой `OPENROUTER_BASE_URL` и моделей, без изменения кода.

## 3. Структура проекта

Используется `src/`‑layout с минимально необходимыми пакетами.

Предлагаемая структура:

- `pyproject.toml` — метаданные проекта и зависимости для `uv`.
- `uv.lock` — lock‑файл зависимостей (генерируется `uv`).
- `Makefile` — команды для локального запуска и работы с Docker.
- `Dockerfile` — образ для локального запуска в контейнере.
- `docs/`
  - `idea.md` — описание продуктовой идеи.
  - `vision.md` — настоящий документ с техническим видением.
  - `tasklist.md` — итерации разработки.
- `src/`
  - `basic_ai_assistant/`
    - `__init__.py`
    - `main.py` — entrypoint приложения (запуск Telegram‑бота).
    - `config.py` — класс конфигурации проекта.
    - `logging_config.py` — инициализация базового логирования.
    - `telegram_bot/`
      - `__init__.py`
      - `bot_app.py` — настройка `aiogram`, обработка текста, фото и команд (`/start`, `/balance`).
    - `llm/`
      - `__init__.py`
      - `llm_client.py` — обёртка над `openai`: structured completion и vision.
      - `schemas.py` — Pydantic‑схемы для structured output.
    - `finance/`
      - `__init__.py`
      - `transaction.py` — dataclass `Transaction`.
      - `transaction_type.py` — enum `TransactionType`.
      - `category.py` — enum `Category` (фиксированный список).
      - `ledger.py` — in-memory хранилище транзакций per user_id.
      - `finance_manager.py` — ядро финансового советника.
    - `dialog/`
      - `__init__.py`
      - `models.py` — `Message`, `UserSession` (контекст диалога).

Все классы располагаются по принципу «1 класс = 1 файл». Enum и dataclass могут находиться в отдельных файлах или рядом с основным классом, если логически связаны и не раздувают модуль.

## 4. Архитектура проекта

Архитектура предельно простая и разделена на четыре зоны:

1. **Telegram‑слой** (`BotApp`)
   - Получает апдейты (текст, фото, команды) через `aiogram`.
   - Скачивает фото чеков через Telegram API.
   - Не содержит бизнес‑логики; делегирует вызовы `FinanceManager`.
2. **Ядро** (`FinanceManager`)
   - Хранит полный контекст диалога для каждого пользователя (`UserSession`).
   - Маршрутизирует: команда `/balance` → расчёт баланса; текст → structured output; фото → VLM.
   - Сохраняет транзакции в `Ledger`, формирует ответ пользователю.
3. **Финансы** (`Ledger`, `Transaction`)
   - In-memory таблица транзакций: один `Ledger` на процесс, записи разделены по `user_id`.
   - Расчёт общего баланса: сумма доходов − сумма расходов.
4. **LLM‑слой** (`LlmClient`, `schemas`)
   - Structured completion для текста (`MODEL_TEXT`).
   - Vision completion для изображений (`MODEL_IMAGE`).
   - Единый endpoint через `OPENROUTER_BASE_URL`.

Высокоуровневый поток данных:

1. Пользователь отправляет текст, фото или команду в Telegram.
2. `BotApp` получает апдейт и вызывает `FinanceManager`.
3. `FinanceManager` обновляет контекст диалога и при необходимости вызывает `LlmClient`.
4. `LlmClient` делает запрос к API (OpenRouter или Ollama) и возвращает structured result.
5. `FinanceManager` сохраняет транзакцию в `Ledger` (если применимо) и формирует ответ.
6. `BotApp` отправляет ответ пользователю в Telegram.

Архитектура намеренно плоская, без сложных паттернов — только разделение ответственности.

## 5. Модель данных

### 5.1. Транзакция (`Transaction`)

Одна запись в таблице учёта:

| Поле | Тип | Описание |
|------|-----|----------|
| `datetime` | `datetime` | Дата и время операции (из сообщения/чека или текущие, если не указано) |
| `direction` | `income` \| `expense` | Доход или расход |
| `amount` | `Decimal` | Сумма в рублях |
| `type` | `TransactionType` | Тип транзакции — определяет LLM по контексту |
| `category` | `Category` | Категория из фиксированного списка |
| `description` | `str` | Подробности: товары, услуги, контрагент, источник дохода и т.п. |

### 5.2. Тип транзакции (`TransactionType`)

- `everyday` — повседневные траты (продукты, кофе, такси).
- `periodic` — периодические платежи (аренда, подписки, коммуналка).
- `one_time` — разовые крупные траты или доходы (покупка техники, премия, подарок).

LLM выбирает тип по контексту сообщения или чека.

### 5.3. Категория (`Category`) — фиксированный список

| Значение | Назначение |
|----------|------------|
| `products` | Продукты |
| `restaurants` | Рестораны, кафе |
| `taxi` | Такси |
| `transport` | Общественный транспорт, бензин |
| `education` | Образование |
| `travel` | Путешествия |
| `health` | Здоровье, аптека |
| `entertainment` | Развлечения |
| `utilities` | Коммунальные услуги, связь |
| `salary` | Зарплата |
| `freelance` | Фриланс, подработка |
| `other` | Прочее (если не подходит ни одна категория) |

### 5.4. Учёт (`Ledger`)

- In-memory: `dict[user_id → list[Transaction]]`.
- Методы: добавить транзакцию, получить список, рассчитать общий баланс.
- Только добавление записей; редактирование и удаление в MVP не предусмотрены.

### 5.5. Контекст диалога

- **Message** — роль (`system` | `user` | `assistant`), текст, опционально timestamp.
- **UserSession** — `user_id`, полная история сообщений диалога (без искусственного усечения на этапе MVP).
- Контекст передаётся в LLM вместе с текущим сообщением для поддержания непрерывного диалога и уточнений.

### 5.6. Хранение

- Транзакции и контекст диалогов хранятся **в памяти процесса**.
- При перезапуске бота данные теряются — допустимо для MVP.
- Персистентность (файлы, БД) — за рамками текущего MVP.

## 6. Работа с LLM

### 6.1. Общие принципы

- Официальный клиент `openai` с `base_url` и ключом из конфигурации.
- **OpenRouter** (разработка): `OPENROUTER_BASE_URL=https://openrouter.ai/api/v1`.
- **Ollama** (продакшен): `OPENROUTER_BASE_URL=http://<gpu-server>:11434/v1`; ключ может быть произвольным (Ollama его не проверяет).
- Две модели: `MODEL_TEXT` (текст, structured output) и `MODEL_IMAGE` (vision для чеков).

### 6.2. Structured output (текст)

`FinanceManager` формирует запрос:

- System prompt — роль финансового советника, правила извлечения, список категорий.
- Полная история диалога пользователя.
- Текущее сообщение.

`LlmClient` вызывает chat completion с `response_format` (JSON Schema из Pydantic) и возвращает `ParsedTextMessage`:

- `intent`: `record_transaction` | `balance_query` | `chat`
- `transaction`: поля `Transaction` (если `record_transaction`)
- `reply`: текст ответа пользователю

### 6.3. Vision (фото чеков)

1. `BotApp` скачивает изображение из Telegram.
2. `LlmClient` отправляет vision‑запрос к `MODEL_IMAGE` с тем же structured output.
3. Схема `ParsedReceiptImage`:
   - `transaction`: поля `Transaction`
   - `reply`: подтверждение или сообщение об ошибке распознавания

### 6.4. Обработка ошибок

- Ошибки LLM логируются.
- Пользователю возвращается безопасное сообщение («Сервис временно недоступен, попробуйте позже»).
- При неудачном распознавании чека — просьба отправить фото повторно или описать трату текстом.

## 7. Сценарии работы

### 7.1. Запись траты/дохода текстом

1. Пользователь: «кофе 350 в starbucks».
2. `FinanceManager` вызывает LLM (structured output).
3. LLM возвращает `intent=record_transaction`, заполненный `transaction`, `reply`.
4. `Ledger` сохраняет транзакцию.
5. Бот отправляет подтверждение: «Записал расход 350 ₽, категория: рестораны…».

### 7.2. Фото чека

1. Пользователь отправляет фото.
2. `BotApp` скачивает файл, передаёт в `FinanceManager`.
3. VLM (`MODEL_IMAGE`) извлекает транзакцию через structured output.
4. `Ledger` сохраняет, бот подтверждает запись.

### 7.3. Запрос баланса

**Явная команда** `/balance`:

1. `BotApp` перехватывает команду, вызывает `FinanceManager.get_balance(user_id)`.
2. `Ledger` считает `sum(income) − sum(expense)`.
3. Бот отвечает: «Ваш баланс: +12 450 ₽».

**Естественная фраза** («какой баланс?»):

1. LLM определяет `intent=balance_query` через structured output.
2. Далее — тот же расчёт и ответ.

### 7.4. Диалог с полным контекстом

1. Пользователь ведёт непрерывный диалог (уточнения, вопросы, несколько записей подряд).
2. `FinanceManager` хранит и передаёт в LLM полную историю сообщений сессии.
3. Бот учитывает предыдущие реплики при интерпретации новых сообщений.

### 7.5. Ошибка при обращении к LLM

1. При запросе к API возникает ошибка (таймаут, неверный ключ, лимит).
2. `LlmClient` логирует ошибку, возвращает информацию в `FinanceManager`.
3. Пользователю — безопасное сообщение об недоступности сервиса.

### 7.6. Ошибка при работе с Telegram API

1. Исключение при отправке/получении логируется.
2. Бот продолжает работу где возможно; при критических ошибках процесс завершается.

## 8. Конфигурирование

Подход максимально простой: все настройки через **переменные окружения**. Для локальной разработки — `.env` с подгрузкой через `python-dotenv`.

### Основные переменные

| Переменная | Описание |
|------------|----------|
| `TELEGRAM_BOT_TOKEN` | Токен Telegram‑бота |
| `OPENROUTER_API_KEY` | API‑ключ OpenRouter или произвольное значение (Ollama) |
| `OPENROUTER_BASE_URL` | Base URL API; по умолчанию `https://openrouter.ai/api/v1` |
| `MODEL_TEXT` | Модель для текста и structured output |
| `MODEL_IMAGE` | Vision‑модель для обработки чеков |
| `LOG_LEVEL` | Уровень логирования (`INFO`, `DEBUG`, `WARNING`) |

Опционально:

- `SYSTEM_PROMPT_PATH` или `SYSTEM_PROMPT` — системная инструкция финансового советника.

### Примеры конфигурации

**Разработка (OpenRouter):**

```
TELEGRAM_BOT_TOKEN=...
OPENROUTER_API_KEY=sk-or-...
OPENROUTER_BASE_URL=https://openrouter.ai/api/v1
MODEL_TEXT=openai/gpt-4o-mini
MODEL_IMAGE=openai/gpt-4o
```

**Продакшен (Ollama на GPU‑сервере):**

```
TELEGRAM_BOT_TOKEN=...
OPENROUTER_API_KEY=ollama
OPENROUTER_BASE_URL=http://gpu-server:11434/v1
MODEL_TEXT=llama3.1:8b
MODEL_IMAGE=llava:13b
```

Переключение провайдера — только сменой `OPENROUTER_BASE_URL` и моделей; код бота не меняется.

Класс `Config` отвечает за чтение переменных, значения по умолчанию и валидацию обязательных параметров.

## 9. Логирование

Для логирования используется стандартный модуль `logging` из Python:

- Инициализация через `logging_config.py`.
- Логи выводятся в stdout (удобно для локальной разработки и Docker).
- `INFO` — старт бота, входящие сообщения, сохранение транзакций.
- `WARNING` — нестандартные, но некритичные ситуации.
- `ERROR` — исключения и проблемы с внешними сервисами (Telegram, LLM).
- Уровень задаётся через `LOG_LEVEL`.

## 10. Сборка и локальный запуск (make + Docker)

### 10.1. Makefile

- `make install` — синхронизация зависимостей через `uv sync`.
- `make run` — локальный запуск (`uv run -m basic_ai_assistant.main`).
- `make docker-build` — сборка Docker‑образа.
- `make docker-run` — запуск контейнера с переменными окружения.

### 10.2. Docker

- Базовый образ: `python:3.11-slim`.
- Установка `uv`, зависимостей, запуск через `uv run -m basic_ai_assistant.main`.
- Переменные окружения передаются через `--env-file` или `-e`.

Таким образом, сборка и запуск остаются простыми; Docker позволяет проверить работу в окружении, близком к продакшену.
