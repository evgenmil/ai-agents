Техническое видение проекта basic-ai-assistant
=============================================

Этот документ описывает минимальное техническое видение проекта Telegram‑бота‑ассистента на базе LLM. Цель — максимально простой MVP для проверки идеи, без оверинжиниринга.

## 1. Технологии, зависимости и сборка

- **Язык**: Python 3.11 (минимальная поддерживаемая версия).
- **Управление зависимостями**: `uv` с использованием `pyproject.toml` и `uv.lock`.
- **Работа с LLM**: официальный клиент `openai`, настроенный на использование провайдера OpenRouter (base URL и ключ берутся из конфигурации/переменных окружения).
- **Интеграция с Telegram**: `aiogram`, только режим `polling` для простоты (без вебхуков и дополнительного веб‑сервера).
- **Локальный запуск без Docker**: `make` вызывает `uv run` с основным entrypoint модуля приложения.
- **Локальный запуск в Docker**: `Dockerfile` на базе `python:3.11-slim`, внутри — установка `uv`, установка зависимостей и запуск того же entrypoint, что и локально.
- **Минимальность стека**: на этапе MVP не используются дополнительные фреймворки, внешние БД, очереди и т.п.; хранение состояния либо в памяти процесса, либо в простых файлах, если потребуется.

## 2. Принципы разработки

- **KISS и минимализм**
  - Каждая новая сущность/абстракция появляется только при реальной необходимости.
  - Предпочитаются простые, прямолинейные решения без лишних слоёв.
- **ООП с принципом 1 класс = 1 файл**
  - Каждый класс располагается в отдельном файле с говорящим именем.
  - Классы небольшие, с понятной и узкой зоной ответственности.
- **Разделение зон ответственности без оверинжиниринга**
  - Модуль/пакет для работы с Telegram (`aiogram` и обработка апдейтов).
  - Модуль/пакет для работы с LLM (формирование запросов к OpenRouter через `openai`).
  - Простое «ядро» диалога, отвечающее за формирование промптов и интерпретацию ответов.
- **Простота тестирования и отладки**
  - Логика по возможности отделена от конкретных клиентов (Telegram, HTTP), чтобы её можно было вызывать напрямую из кода или CLI.
  - Тесты не являются обязательными на первом шаге, но код пишется так, чтобы их можно было легко добавить.
- **Явная конфигурация**
  - Секреты (ключ OpenRouter, токен Telegram) хранятся только в переменных окружения.
  - Поверх окружения может быть один простой класс/модуль конфигурации, инкапсулирующий чтение настроек.

## 3. Структура проекта

Используется `src/`‑layout с минимально необходимыми пакетами.

Предлагаемая структура:

- `pyproject.toml` — метаданные проекта и зависимости для `uv`.
- `uv.lock` — lock‑файл зависимостей (генерируется `uv`).
- `Makefile` — команды для локального запуска и работы с Docker.
- `Dockerfile` — образ для локального запуска в контейнере.
- `docs/`
  - `idea.md` — описание продуктовой идеи.
  - `vision.md` — настоящий документ с техническим видением.
- `src/`
  - `basic_ai_assistant/`
    - `__init__.py`
    - `main.py` — entrypoint приложения (запуск Telegram‑бота).
    - `config.py` — класс конфигурации проекта.
    - `logging_config.py` — инициализация базового логирования.
    - `telegram_bot/`
      - `__init__.py`
      - `bot_app.py` — класс, инкапсулирующий настройку `aiogram` и запуск polling.
    - `llm/`
      - `__init__.py`
      - `llm_client.py` — класс‑обёртка над `openai`/OpenRouter.
    - `dialog/`
      - `__init__.py`
      - `dialog_manager.py` — класс‑ядро диалога (управление контекстом и промптами).
      - `models.py` — простые классы/датаклассы модели данных диалога (сообщения, сессии и т.п.).

Все классы располагаются по принципу «1 класс = 1 файл». Если в модуле нужны дополнительные типы (enum, dataclass), они могут находиться в том же файле, если логически относятся к основному классу и не раздувают модуль.

## 4. Архитектура проекта

Архитектура предельно простая и разделена на три основных слоя:

1. **Telegram‑слой**
   - Отвечает за получение апдейтов от Telegram (через `aiogram`) и отправку ответов пользователю.
   - Не содержит бизнес‑логики диалога; только преобразует апдейты в вызовы `DialogManager`.
2. **Ядро диалога**
   - Представлено классом `DialogManager`.
   - Хранит минимальный контекст диалога (например, несколько последних сообщений).
   - Формирует запросы к LLM (список сообщений с ролями: system / user / assistant).
   - Получает ответ от LLM и возвращает текст ответа Telegram‑слою.
3. **LLM‑слой**
   - Представлен классом `LlmClient`.
   - Инкапсулирует работу с официальным клиентом `openai`, настроенным на OpenRouter.
   - Знает про базовый URL, модель и параметры запроса (temperature и т.д.), получая их из конфигурации.

Высокоуровневый поток данных:

1. Пользователь пишет сообщение в Telegram‑чате.
2. `aiogram` через `BotApp` получает апдейт и вызывает метод `DialogManager` (например, `handle_user_message`).
3. `DialogManager` обновляет контекст диалога, формирует список сообщений для LLM и вызывает `LlmClient`.
4. `LlmClient` делает запрос к OpenRouter через `openai`, возвращает текст ответа.
5. `DialogManager` может при необходимости скорректировать/обогатить ответ и возвращает его в `BotApp`.
6. `BotApp` отправляет ответ пользователю в Telegram.

Архитектура намеренно плоская и небольшая, без сложных слоёв, интерфейсов и паттернов — только самое необходимое для разделения ответственности и возможного дальнейшего расширения.

## 5. Модель данных

Для MVP используются только самые простые сущности, необходимые для диалогов:

- **Message**
  - Роль: `system` | `user` | `assistant`.
  - Текст сообщения.
  - Опционально: отметка времени.
- **UserSession**
  - Идентификатор пользователя (Telegram user id).
  - Список последних сообщений диалога (например, ограниченный по длине).
  - Опциональные настройки пользователя (например, выбранная роль ассистента или тон, если это понадобится).
- **DialogContext**
  - Объединяет `UserSession` и системный промпт.
  - Упрощённая обёртка поверх данных, которые передаются в LLM.

На этапе MVP:

- Состояние диалога хранится **в памяти процесса** в простых структурах (словарь id пользователя → `UserSession`).
- При перезапуске бота история диалогов может теряться — это допустимо для первичной проверки идеи.
- Если позже потребуется персистентность, можно будет добавить файловое или внешнее хранилище, не меняя базовую модель данных.

## 6. Работа с LLM

Основные принципы:

- Используется официальный клиент `openai`, но настроенный на работу с OpenRouter (через `base_url` и API‑ключ OpenRouter).
- API‑ключ OpenRouter и идентификатор модели задаются через конфигурацию/переменные окружения.
- Вызов модели — это обычный чат‑запрос (chat completion) с сообщениями в формате: system + несколько последних пар user/assistant + текущее пользовательское сообщение.

Базовый поток:

1. `DialogManager` формирует список сообщений:
   - Один базовый **system prompt**, определяющий стиль и роль ассистента.
   - Последние сообщения пользователя и ассистента для сохранения контекста.
   - Текущее пользовательское сообщение.
2. `LlmClient` делает вызов к выбранной модели OpenRouter через клиент `openai`.
3. В случае успеха возвращает текст ответа.
4. В случае ошибки:
   - Логирует ошибку.
   - Возвращает в `DialogManager` информацию об ошибке, чтобы тот мог выдать пользователю простое сообщение вида «Сервис временно недоступен, попробуйте позже».

Минимальные настройки:

- Идентификатор модели (например, одна конкретная модель OpenRouter) берётся из конфигурации.
- Параметры вроде `temperature`, `max_tokens` и т.п. задаются один раз в конфиге и редко меняются.

## 7. Сценарии работы

### 7.1. Базовый сценарий (happy path)

1. Разработчик настраивает переменные окружения (как минимум токен Telegram и ключ OpenRouter).
2. Запускается бот (`make run` или соответствующий Docker‑контейнер).
3. Пользователь пишет сообщение боту в Telegram.
4. Бот получает апдейт, `DialogManager` формирует запрос к LLM.
5. LLM возвращает ответ, бот отправляет его пользователю.
6. Пользователь продолжает диалог; контекст поддерживается за счёт хранения нескольких последних сообщений.

### 7.2. Ошибка при обращении к LLM

1. При запросе к OpenRouter возникает ошибка (сетевой таймаут, неверный ключ, лимит и т.д.).
2. `LlmClient` логирует ошибку и возвращает информацию об ошибке в `DialogManager`.
3. `DialogManager` отдаёт в Telegram‑слой безопасное пользовательское сообщение («Сервис временно недоступен…»).
4. Логи позволяют разработчику разобраться в проблеме.

### 7.3. Ошибка при работе с Telegram API

1. При отправке сообщения или получении апдейта возникает исключение.
2. Ошибка логируется.
3. Там, где это возможно, бот продолжает работу; при критических ошибках процесс завершается, и разработчик перезапускает бота.

## 8. Конфигурирование

Подход к конфигурации максимально простой:

- Все секреты и чувствительные данные задаются только через **переменные окружения**.
- Для локальной разработки допускается использование `.env`‑файла (при желании) с подгрузкой переменных через внешний инструмент, но приложение само по себе может не зависеть от `python-dotenv`.

Основные переменные окружения:

- `TELEGRAM_BOT_TOKEN` — токен Telegram‑бота.
- `OPENROUTER_API_KEY` — ключ доступа к OpenRouter.
- `OPENROUTER_BASE_URL` — базовый URL API OpenRouter (по умолчанию значение по документации OpenRouter).
- `LLM_MODEL` — идентификатор модели LLM по умолчанию.
- `LOG_LEVEL` — уровень логирования (например, `INFO`, `DEBUG`, `WARNING`).
- Опционально: `SYSTEM_PROMPT_PATH` или `SYSTEM_PROMPT` для задания системной инструкции (system prompt) из файла или строки.

Класс конфигурации (`Config`) отвечает за:

- Чтение нужных переменных окружения.
- Применение значений по умолчанию, где это уместно.
- Валидацию наличия критичных параметров (токены, ключи).

## 9. Логирование

Для логирования используется стандартный модуль `logging` из Python:

- Инициализация через один модуль `logging_config.py` (функция, настраивающая `basicConfig`).
- Логи выводятся в stdout, что удобно для локальной разработки и Docker.
- Минимальный набор уровней:
  - `INFO` — основные события (старт бота, входящие сообщения, ответы LLM по желанию).
  - `WARNING` — нестандартные, но некритичные ситуации.
  - `ERROR` — исключения и проблемы с внешними сервисами (Telegram, LLM).
- Уровень логирования задаётся через переменную окружения `LOG_LEVEL`.

Никакой сложной инфраструктуры логирования (ELK, внешние сервисы) на этапе MVP не используется.

## 10. Сборка и локальный запуск (make + Docker)

### 10.1. Makefile

Makefile предоставляет минимальный набор целей:

- `make install` — установка/синхронизация зависимостей через `uv` (например, `uv sync`).
- `make run` — локальный запуск бота без Docker (например, `uv run -m basic_ai_assistant.main`).
- `make lint`/`make test` — опциональные команды для проверки кода, если они будут добавлены позже.
- `make docker-build` — сборка Docker‑образа.
- `make docker-run` — запуск контейнера с пробросом необходимых переменных окружения.

Все команды ориентированы на простейший сценарий разработки и не предполагают сложных пайплайнов CI/CD.

### 10.2. Docker

Docker используется только для **локального запуска в контейнере**, без деплоя на удалённые серверы:

- Базовый образ: `python:3.11-slim`.
- В Dockerfile:
  - Установка системных зависимостей, необходимых для `uv` и Python‑пакетов.
  - Установка/скачивание `uv`.
  - Копирование файлов проекта (`pyproject.toml`, `uv.lock`, `src/`, `docs/`, `Makefile` при необходимости).
  - Установка зависимостей через `uv`.
  - Команда запуска по умолчанию: запуск бота через `uv run -m basic_ai_assistant.main` или через `make run`.

Запуск контейнера предполагает передачу токена Telegram и ключа OpenRouter через переменные окружения (или `--env-file` для локальной разработки).

Таким образом, сборка и запуск остаются простыми и прозрачными, а Docker даёт возможность проверить работу бота в окружении, близком к продакшену, без фактического деплоя.
